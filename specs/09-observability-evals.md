# Spec 09 — Phase 8: Observability, Evals & Cost Modeling

**Status:** Ready
**Branch:** `phase-8-observability-evals`
**Depends on:** Phase 6, Phase 7

## Context

This is the phase that turns a demo into something defensible to a client — measurable quality (evals) and measurable cost, both derived from real data, not estimates.

## Functional Requirements

1. `testCases.ts` in `/backend/src/evals/` — at least 20 cases, each:
   ```ts
   type EvalCase = {
     query: string;
     mustInclude: string[]; // listing ids/titles expected in top 5
     mustExclude: string[]; // listings that must NOT appear (e.g. non-pet-friendly for a pet-friendly query)
   }
   ```
   - Must cover: explicit filter queries, synonym/paraphrase cases (query wording differs from listing wording), subjective/subtle-intent queries, and at least 3 cases specifically targeting the long-tail listings from Phase 1 to confirm they surface when actually relevant.
2. `runEvals.ts` — runs each case through the real `/api/search` endpoint (not a unit-level mock — this should exercise the full pipeline), scores pass/fail against `mustInclude`/`mustExclude`, prints a summary table and overall pass rate. Exposed as `pnpm --filter backend run eval`.
3. Langfuse integration: every Claude API call across the pipeline (extraction, query understanding, rerank) traced through Langfuse — prompt, response, latency, token counts — each trace tagged with its pipeline stage name.
4. `costModel.ts` — takes observed average token usage per stage (from Langfuse traces or `search_logs`) and current Anthropic pricing, outputs estimated $/query and $/1000-searches at 3 traffic tiers: 100/day, 1,000/day, 10,000/day.
5. All of the above summarized into `EVAL_REPORT.md`, generated by the scripts (not hand-written), containing: eval pass rate + per-case detail, average latency per stage, and the cost model table.

## Interfaces

```
pnpm --filter backend run eval    → runs eval suite, writes EVAL_REPORT.md
pnpm --filter backend run cost-model → appends/updates cost table in EVAL_REPORT.md
```

## Non-Functional Requirements

- Cost model numbers must be traceable to actually-observed token counts, not assumed averages — document the data source used.

## Explicit Out of Scope

- No automated regression gating in CI yet (that's a natural Phase 10/CI extension, not required here).
- No UI for viewing eval results (markdown report is sufficient).

## Acceptance Criteria

- [ ] `pnpm --filter backend run eval` runs all 20+ cases against the live pipeline and produces a pass rate with per-case pass/fail detail.
- [ ] At least one long-tail listing case passes (confirms niche listings are actually reachable, not just theoretically).
- [ ] Langfuse dashboard shows a complete trace for a sample search session, with all three LLM call stages visible and correctly tagged.
- [ ] `EVAL_REPORT.md` contains a cost table with real numbers (not placeholders) for all 3 traffic tiers.
- [ ] Re-running the eval suite twice produces a stable pass rate (allowing for normal LLM output variance — flag and investigate if any case flips pass/fail between runs for reasons other than genuine ambiguity).

## Open Questions Claude Code Should Ask If Unclear

- Acceptable pass-rate threshold to consider the pipeline "good enough" to proceed to Phase 9 — this is a product judgment call, not something to decide silently.
